## Функціональність вебзастосунку

### 1. Модуль авторизації та автентифікації
- **Вхід в систему**: вход з email та паролем
- **Управління користувачами**: реєстрація нових користувачів
- **Ролі користувачів**: USER, ADMIN
- **Безпека**: використання JWT токенів, хешування паролів (bcryptjs)

### 2. Модуль управління клієнтами (Customers)
**Функції:**
- Додавання нових клієнтів з контактною інформацією
- Ведення історії всіх автомобілів клієнта
- Зберігання історії комунікацій з клієнтом
- Додавання особистих нотаток про клієнта
- Пошук клієнтів за ім'ям, телефоном або номером авто
- Архівування неактивних клієнтів
- Перегляд деталей клієнта та його історії обслуговування

**Структура даних клієнта:**
```
Клієнт
├── ID, Ім'я, Телефон
├── Джерело (звідки дізнався про СТО)
├── Статус (активний/архівований)
├── Список автомобілів
├── Історія комунікацій
├── Особисті нотатки
└── Замовлення
```

### 3. Модуль управління автомобілями
**Функції:**
- Додавання автомобілів до клієнта
- Збереження даних: марка, модель, номер, VIN
- Перегляд історії обслуговування авто
- Видалення автомобілів
- Привʾязка замовлень до конкретного авто

### 4. Модуль управління замовленнями (Dashboard)
**Функції:**
- Створення нових замовлень на обслуговування
- Вибір послуг та запчастин
- Назначення майстра
- Установка терміну виконання
- Управління статусом замовлення:
  - PENDING (очікування)
  - IN_WORK (в роботі)
  - READY (готово)
  - COMPLETED (завершено)
  - CANCELLED (скасовано)
- Розрахунок вартості замовлення
- Перегляд поточних та архівованих замовлень
- Управління сплатою (готівка/інше)
- Екстрені замовлення (з позначкою)

### 5. Модуль управління послугами (Services)
**Функції:**
- Управління каталогом послуг
- Установка ціни та старої ціни (для скидок)
- Категоризація послуг
- Додавання рекомендацій до послуг
- Виведення часу виконання
- Підбір запчастин до послуги
- Пошук та фільтрація послуг
- Друк каталогу послуг

**Типи послуг:**
- Діагностика
- ТО (технічне обслуговування)
- Ремонт
- Кузовні роботи
- Хімчистка
- Шиномонтажні роботи
- Інші послуги

### 6. Модуль управління складом (Inventory)
**Функції:**
- Облік запчастин та матеріалів
- Категоризація товарів
- Управління кількістю (наявне/мінімум)
- Управління постачальниками
- Логування руху товарів (приход/видаток)
- Установка мінімального залишку
- Технічні параметри товарів
- Сумісність з автомобілями
- Предупереждения про низькі запаси

**Статус товарів:**
- Достатня кількість
- Потребує замовлення (нижче мінімуму)
- Недоступно

### 7. Модуль управління персоналом (Staff)
**Функції:**
- Управління списком майстрів
- Категоризація персоналу (механік, електрик, м'яка обробка)
- Ведення досвіду роботи
- Статус майстра (вільний/в роботі)
- Розрахунок заробітку та комісій
- Логування дій персоналу
- Текучі роботи майстра
- Видалення з системи (архівування)

### 8. Модуль аналітики (Analytics)
**Функції:**
- Статистика за період
- Графіки доходу
- Топ послуг за попитом
- Рейтинг майстрів (за виконаними замовленнями)
- Середня вартість замовлення
- Кількість замовлень за період
- Популярні послуги

**Метрики:**
- Загальний дохід
- Кількість замовлень
- Середня вартість
- Рейтинг сервісів
- Розподіл послуг

### 9. Модуль сповіщень (Notifications)
**Функції:**
- Автоматичні сповіщення:
  - Низькі запаси на складі
  - Сповіщення про послуги
  - Затримки виконання замовлень
- Ручні сповіщення
- Пріоритизація (HIGH, MEDIUM, LOW)
- Позначення як прочитаних
- Видалення сповіщень

### 10. Модуль керування замовленнями на закупівлю
**Функції:**
- Створення замовлень на закупівлю
- Управління позиціями в замовленні
- Відстеження статусу (Pending/Received)
- Розрахунок вартості

## Алгоритм роботи застосунку

### 1. Процес запуску застосунку

```
СТАРТ
  │
  ├─► Ініціалізація браузера
  │
  ├─► Завантаження React компонентів
  │
  ├─► Підключення до API (бекенд)
  │
  ├─► Перевірка наявності JWT токена в localStorage
  │    │
  │    ├─► ЯКЩО токен є ─► Верифікація токена на сервері
  │    │                   │
  │    │                   ├─► ЯКЩО валідний ─► Перехід на Dashboard
  │    │                   │
  │    │                   └─► ЯКЩО невалідний ─► Видалення токена
  │    │
  │    └─► ЯКЩО токена немає ─► Перехід на Login
  │
  └─► ГОТОВО
```

### 2. Процес авторизації

```
LOGIN ФОРМА
  │
  ├─► Введення Email та Password
  │
  ├─► ВАЛІДАЦІЯ на клієнті
  │    ├─► Email формат
  │    └─► Пароль не пустий
  │
  ├─► Відправлення запиту на сервер
  │    POST /auth/login
  │
  ├─► СЕРВЕР: Пошук користувача в БД
  │    │
  │    ├─► ЯКЩО користувач не знайдено
  │    │    └─► Повернення помилки 404
  │    │
  │    └─► ЯКЩО користувач знайдено
  │         │
  │         ├─► Порівняння пароля з хешем
  │         │
  │         ├─► ЯКЩО пароль невірний
  │         │    └─► Повернення помилки 401
  │         │
  │         └─► ЯКЩО пароль вірний
  │              │
  │              ├─► Оновлення lastLoginAt
  │              │
  │              ├─► Генерація JWT токена
  │              │
  │              └─► Повернення токена та даних користувача
  │
  ├─► КЛІЄНТ: Збереження токена в localStorage
  │
  ├─► Перенаправлення на Dashboard
  │
  └─► ГОТОВО
```

### 3. Процес створення замовлення

```
DASHBOARD - СТВОРЕННЯ ЗАМОВЛЕННЯ
  │
  ├─► Натискання кнопки "+ Нове замовлення"
  │
  ├─► Відкриття модального вікна (OrderModal)
  │
  ├─► ФОРМА:
  │    ├─► Вибір клієнта (або створення нового)
  │    ├─► Вибір автомобіля
  │    ├─► Вибір послуг (можна кілька)
  │    ├─► Автоматичний розрахунок вартості
  │    ├─► Назначение майстра
  │    ├─► Установка терміну
  │    ├─► Вибір способу оплати
  │    ├─► Позначка "Екстрене" (опціонально)
  │    └─► Введення примітки
  │
  ├─► ВАЛІДАЦІЯ
  │    ├─► Клієнт вибран
  │    ├─► Послуга вибрана
  │    └─► Майстер вибран
  │
  ├─► Натискання "Зберегти"
  │
  ├─► СЕРВЕР: Збереження замовлення в БД
  │    │
  │    ├─► Створення запису Order
  │    ├─► Додавання послуг в Order.services
  │    ├─► Використання запчастин (decrease Inventory)
  │    ├─► Логування на сервер
  │    └─► Генерація сповіщення якщо потрібно
  │
  ├─► КЛІЄНТ: Оновлення списку замовлень
  │
```

## 9. Загальний опис обраної архітектури

### Принципи архітектури

Вебзастосунок STO (Система управління станцією технічного обслуговування) розроблений за принципом **«клієнт-сервер»**, де користувач взаємодіє з програмою через звичайний браузер. Таке програмне забезпечення дозволяє адміністраторам та персоналу СТО працювати з даними з будь-якої точки світу. Це стає можливим завдяки поєднанню мережі, серверного обладнання та програм, які дозволяють користувачу та серверу обмінюватися інформацією на відстані.

### Модель клієнт-серверної архітектури

Клієнт-серверна архітектура STO-застосунку працює за принципом розподілу відповідальності:

```
┌─────────────────────────────────────────────────────────────┐
│                    КЛІЄНТСЬКА ЧАСТИНА                        │
│  (React браузер - користувацький інтерфейс, UX)             │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  Браузер користувача на ПК / планшеті                        │
│  ↓                                                            │
│  React 19 + Context API (управління станом)                 │
│  ↓                                                            │
│  Компоненти інтерфейсу (сторінки, модальні вікна)           │
│  ↓                                                            │
│  Axios (HTTP клієнт)                                         │
│                                                               │
└─────────────────────────────────────────────────────────────┘
              │
              │ HTTP запити (GET, POST, PUT, DELETE)
              │ + JWT токен в заголовках
              │ JSON формат даних
              ▼
┌─────────────────────────────────────────────────────────────┐
│                    СЕРВЕРНА ЧАСТИНА                          │
│  (Node.js/Express - бізнес-логіка, обробка даних)          │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  Сервер (хостинг, compute resources)                        │
│  ↓                                                            │
│  Express.js (маршрутизація та обробка запитів)             │
│  ↓                                                            │
│  Middleware (автентифікація, валідація, логування)         │
│  ↓                                                            │
│  Controllers (обробка бізнес-логіки)                       │
│  ↓                                                            │
│  Prisma ORM (взаємодія з БД)                                │
│                                                               │
└─────────────────────────────────────────────────────────────┘
              │
              │ SQL запити
              │ ACID транзакції
              │ Реляційні дані
              ▼
┌─────────────────────────────────────────────────────────────┐
│                    БАЗА ДАНИХ                                │
│  (SQLite - зберігання та управління даними)                 │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  SQLite База даних (dev.db)                                  │
│  ├─ User, Customer, Car, Order                              │
│  ├─ Service, Inventory, Staff                               │
│  ├─ Notification, StaffLog, InventoryLog                    │
│  └─ [12+ таблиць]                                           │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

### Переваги такої архітектури

1. **Розділення обов'язків**: серверна та клієнтська частини існують окремо, тому程式員 не потрібно дублювати одну й ту саму логіку в обох місцях. Вся бізнес-логіка зосереджена на сервері.

2. **Безпека даних**: оскільки всі дані зберігаються на сервері (SQLite база), а не на пристрої користувача, їх набагато легше захистити від крадіжки та несанкціонованого доступу.

3. **Централізована автентифікація**: сервер сам вирішує, кому дозволити перегляд даних через JWT токени та bcrypt хешування паролів, що спрощує контроль за безпекою.

4. **Економічність**: обслуговування однієї централізованої SQLite бази даних зазвичай дешевше, ніж спроби синхронізувати дані між тисячами окремих пристроїв.

5. **Доступність**: адміністратори можуть працювати з системою з будь-якого браузера та з будь-якої географічної точки.

6. **Масштабованість**: архітектура дозволяє легко додавати нові функції та модулі без порушення існуючого коду.

### Недоліки клієнт-серверної архітектури

1. **Залежність від з'єднання**: головний мінус у тому, що без стабільного з'єднання із сервером програма просто не зможе працювати. Потрібен постійний доступ до інтернету.

2. **Затримки мережі**: всі операції вимагають круглої подорожі запит-відповідь, що може бути повільнішим ніж локальні обчислення.

3. **Навантаження на сервер**: централізація означає, що сервер повинен бути достатньо потужним для обслуговування всіх одночасних запитів.

### Модель обміну даними

Основу роботи клієнт-серверної моделі складає постійний обмін HTTP-повідомленнями: користувач відправляє запит, а сервер готує та повертає відповідь у форматі JSON.

#### Структура HTTP-запиту від клієнта:

```
HTTP запит складається з таких частин:

┌───────────────────────────────────────────────────────────┐
│ Рядок запиту (Request Line)                               │
├───────────────────────────────────────────────────────────┤
│ Метод     URL                         Версія HTTP         │
│ --------- ----------------------------- --------         │
│ POST      /api/customers/create        HTTP/1.1          │
│                                                            │
│ GET       /api/orders?status=PENDING    HTTP/1.1          │
│                                                            │
│ PUT       /api/inventory/5/update       HTTP/1.1          │
│                                                            │
│ DELETE    /api/staff/3                  HTTP/1.1          │
└───────────────────────────────────────────────────────────┘
```

**Метод (HTTP verb)**: вказує, що саме ми хочемо зробити:
- **GET** — просто подивитися дані (без змін на сервері)
- **POST** — створити щось нове (нового клієнта, замовлення)
- **PUT** — змінити старе (оновити дані)
- **DELETE** — видалити

**Версія протоколу**: зазвичай це стандартні HTTP/1.1 або сучасніший HTTP/2.

**URL-адреса (Host)**: точна «координата» на сервері, куди саме спрямований запит. У STO-проекті приклади:
- `/api/customers` — список клієнтів
- `/api/orders/123` — конкретне замовлення
- `/api/inventory/search?name=battery` — пошук за номером

**Заголовки (Headers)**: технічні мітки, де вказується:
- `Content-Type: application/json` — тип даних (JSON)
- `Authorization: Bearer <JWT_TOKEN>` — токен для автентифікації
- `Accept-Language: uk-UA` — мова інтерфейсу

Приклад заголовків у STO-запиті:
```
POST /api/orders/create HTTP/1.1
Host: localhost:5000
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Content-Length: 245
```

**Тіло запиту (Body)**: це самі дані (наприклад, інформація про замовлення), які ми передаємо на сервер. У STO-системі дані зазвичай оформлені у форматі JSON:

```json
{
  "customerId": 5,
  "carId": 12,
  "services": [1, 3, 7],
  "masterId": 2,
  "totalPrice": 1500,
  "paymentMethod": "cash",
  "isUrgent": false,
  "notes": "Замінити масло та фільтр"
}
```

#### Структура HTTP-відповіді від сервера:

Після того, як сервер отримує запит від користувача, він надсилає зворотне повідомлення — HTTP-відповідь. Її структура дозволяє браузеру миттєво зрозуміти, чи вдалося виконати дію.

```
┌───────────────────────────────────────────────────────────┐
│ Рядок статусу (Status Line)                               │
├───────────────────────────────────────────────────────────┤
│ Версія    Код        Текстове пояснення                   │
│ --------- -------- ----------------                      │
│ HTTP/1.1  201      Created                                │
│                                                            │
│ HTTP/1.1  200      OK                                     │
│                                                            │
│ HTTP/1.1  400      Bad Request                            │
│                                                            │
│ HTTP/1.1  401      Unauthorized                           │
│                                                            │
│ HTTP/1.1  404      Not Found                              │
│                                                            │
│ HTTP/1.1  500      Internal Server Error                  │
└───────────────────────────────────────────────────────────┘
```

**Код статусу**: спеціальне тризначне число, що є головним індикатором результату:

- **1xx (Інформаційні)**: запит отримано, сервер ще працює над ним
  - 100 Continue — продовжуйте надсилання даних
  - 101 Switching Protocols — зміна протоколу

- **2xx (Успіх)**: все пройшло чудово
  - 200 OK — дані успішно отримані та оброблені
  - 201 Created — новий запис успішно створений (нового клієнта, замовлення)
  - 204 No Content — запит успішно, але немає даних для повернення (при DELETE)

- **3xx (Перенаправлення)**: ресурс переміщений
  - 301 Moved Permanently — посилання змінилось назавжди
  - 302 Found — тимчасове перенаправлення

- **4xx (Помилка користувача)**: проблема в самому запиті
  - 400 Bad Request — неправильний формат запиту
  - 401 Unauthorized — відсутній або невірний JWT токен
  - 403 Forbidden — у користувача немає прав доступу
  - 404 Not Found — клієнт не знайдено, замовлення не існує
  - 409 Conflict — конфлікт даних (наприклад, дублювання)

- **5xx (Помилка сервера)**: збій стався всередині програми
  - 500 Internal Server Error — непередбачена помилка на сервері
  - 503 Service Unavailable — сервер тимчасово недоступний

**Заголовки (Headers)**: додаткові мітки про саму відповідь:
- `Content-Type: application/json` — формат повернутих даних
- `Date: Wed, 10 May 2026 14:30:00 GMT` — час відправлення
- `Set-Cookie: session=...` — встановлення cookies

**Тіло відповіді (Body)**: це те, що ми власне чекаємо від сервера. Приклади відповідей у STO-системі:

Успішна відповідь (201 Created):
```json
{
  "success": true,
  "message": "Замовлення успішно створено",
  "data": {
    "id": 156,
    "customerId": 5,
    "status": "PENDING",
    "totalPrice": 1500,
    "createdAt": "2026-05-10T14:30:00Z"
  }
}
```

Помилка (400 Bad Request):
```json
{
  "success": false,
  "message": "Не всі обов'язкові поля заповнені",
  "errors": {
    "customerId": "Клієнт не вибран",
    "services": "Повинна бути хоча б одна послуга"
  }
}
```

Помилка автентифікації (401 Unauthorized):
```json
{
  "success": false,
  "message": "Невірний або відсутній JWT токен",
  "code": "AUTH_FAILED"
}
```

```
4. КЛІЄНТ (React) отримує відповідь:
   ═════════════════════════════════════════════════════════════
   
   ✓ Парсить JSON
```

### Технологічні характеристики протоколу

| Аспект | Опис |
|--------|------|
| **Версія HTTP** | HTTP/1.1 (стандартна), можливість HTTP/2 (швидше) |
| **Шифрування** | HTTPS (в production), HTTP (development) |
| **Формат даних** | JSON (всі API endpoints) |
| **Автентифікація** | JWT токени в заголовку `Authorization: Bearer <token>` |
| **Кодування** | UTF-8 (підтримка українських символів) |
| **Хешування паролів** | bcryptjs (8+ символів) |
| **Таймаути** | Стандартні HTTP таймаути браузера (~30 сек) |
| **CORS** | Налаштовано для кросс-доменних запитів |
| **Логування** | Усі запити логуються на сервері (операції, помилки) |

### Інтеграція компонентів системи

```
БРАУЗЕР ◄──────► EXPRESS SERVER ◄──────► SQLite БД
  ↓                    ↓                      ↓
React Components   Routes/Controllers    20+ таблиці
  ↓                    ↓                      ↓
Axios HTTP Calls  Middleware Stack        Prisma ORM
  ↓                    ↓                      ↓
Context API        Validators             Транзакції
  ↓                    ↓                      ↓
Local Storage      Business Logic         Реляційні дані
```

**Flux даних:**
1. Користувач взаємодіє з React компонентом
2. React відправляє Axios HTTP запит на сервер
3. Express обробляє запит через middleware та контролери
4. Сервер виконує запит до SQLite через Prisma ORM
5. БД повертає дані або статус операції
6. Сервер форматує JSON відповідь
7. React отримує дані та оновлює UI

Така архітектура забезпечує стабільну, безпечну та масштабовану систему управління СТО.
